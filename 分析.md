先说结论：**MMAUD 的结果基本合理，不用慌；AIQII 的结果不正常，有明显的数据/标签层面的问题，而且几乎可以锁定不是模型代码的问题。** 下面拆开讲，最后给你一个明确的排查顺序。

我先快速查一下 AIQII 是不是公开数据集、有没有公开的标注规范，这会直接影响怎么查。没找到叫 AIQII 的公开数据集，所以它大概率是你自己的/内部数据集。不过没关系——诊断不靠它的来源，靠的是数字内部的"自相矛盾"。下面把两个数据集分别给你判一下。

## 一、MMAUD：合理，别慌

MMAUD 这几张表（结果 A/B、扩展套件）整体落在"毫米波雷达检测小无人机"的正常区间：

* `Center AP@2m` 普遍 50~97%、中心误差约 1~2m → 检出和粗定位没问题。
* `3D AP@0.25` 从 20% 到 89%（PFANet 结果 A 到 89.79%），这对雷达来说**已经算不错**了。
* `3D AP@0.5` 普遍 <10% → 这是雷达的"通病"：点太稀，框的尺寸/朝向回归不准，到不了 IoU 0.5。**这是模态的物理极限，不是你的错。**

所以 MMAUD 不需要怀疑数据或代码，它就是"雷达本该的样子"。唯一要收尾的是我上次提的 **结果 A vs 结果 B 不一致** （同一个 run tag 数值差很多，CenterPoint 的 Center AP 差 64 个点），这个得确定哪一份是最终版，别两份混着汇报。

## 二、AIQII：不正常，问题几乎可以锁定在数据/标签

AIQII 这两行（OpenPCDet / InterFusion）有一个 **自相矛盾的特征** ，这才是真正的红旗：

| 指标         |                                AIQII | MMAUD（同模型，结果A） |
| ------------ | -----------------------------------: | ---------------------: |
| Center AP@2m |            **94.21% / 93.41%** |        96.48% / 93.81% |
| 平均中心误差 | **0.521m / 0.623m** （更好！） |        1.124m / 1.135m |
| BEV AP@0.5   |              **7.68% / 7.78%** |        66.03% / 65.29% |
| 3D AP@0.25   |              **9.19% / 8.11%** |        68.60% / 77.85% |
| 3D AP@0.5    |              **0.20% / 0.20%** |          7.33% / 9.54% |

看出问题了吗：**AIQII 上模型把中心点定得比 MMAUD 还准（误差只有 0.52m），但凡是涉及"框重叠"的指标（BEV/3D AP）几乎全部归零。**

这是一个逻辑上很硬的判断：

* 如果 AIQII 只是"雷达更难、点更稀"，那中心误差应该更大才对——可它反而更小（0.52m vs 1.1m）。说明 AIQII 的点云其实给了 **很好的位置信息** ，模型也学到了。
* 既然中心几乎完美，但连最宽松的 `3D AP@0.25`（只要 25% 重叠）都只有 9%，那只剩一种解释： **预测框和真值框的尺寸 / 朝向 / 高度系统性地对不上** 。中心对、框对不上 = 框的"定义"出了问题。

而 BEV AP@0.5 也只有 7%（BEV 不看高度 z），说明问题不只是 z 轴，连俯视的长宽框本身都对不齐。**这几乎一定是 AIQII 的标注规范 / 框尺寸 / 坐标系 / anchor 配置层面的问题，不是模型架构问题。**

## 三、为什么能断定不是模型代码？

因为 **同一套模型代码（OpenPCDet、InterFusion），在 MMAUD 上结果正常且自洽，在 AIQII 上却塌掉** 。区分变量只有"数据集"这一个。如果是检测器/损失函数/评估脚本坏了，MMAUD 应该出现同样的病症——但它没有。所以问题出在  **AIQII 这条数据链路上** （数据加载、标签转换、anchor 配置），而不是模型本身。

## 四、接下来怎么办——按这个顺序查 AIQII

**先查数据集，别动模型架构。** 具体五步，从快到慢：

1. **直接可视化（最快，10 分钟见分晓）** ：随便挑 3~5 帧 AIQII，把雷达点 + 真值框（GT）+ 预测框画在一张图里。十有八九你会看到"框的大小/朝向明显不对"或"GT 框本身就异常"。这一步基本能当场定位。
2. **打印 GT 框尺寸分布** ：统计 AIQII 真值框的长宽高（l/w/h）直方图，和 MMAUD 对比。无人机框应该在 0.2~1.5m 量级。如果 AIQII 的框是厘米级、零、或异常大，IoU 必然崩。顺带确认单位是不是米（有没有混进厘米/毫米的缩放）。
3. **核对 anchor 配置** ：你给 AIQII 用的 anchor 尺寸是不是按 AIQII 的真实框大小设的？如果直接套了 MMAUD 的 anchor、或没改默认的车辆 anchor（3.9×1.6×1.5m），回归目标会整体跑偏，正好造成"中心准、框崩"。
4. **检查朝向（yaw）约定** ：弧度 vs 角度、有没有 90° 偏移或轴翻转。朝向错会让框旋转，IoU 直接掉。
5. **检查 GT 框内点数 / 评估范围** ：如果 AIQII 很多真值框里radar 点数接近 0（标注范围和点云范围没对齐，或时间没同步），AP 也会塌。看一下 min points per GT。

另外值得查一下  **AIQII 的标签是怎么生成的** ——如果是用别的传感器/跟踪器自动生成的伪标签，框尺寸不可靠是很常见的根因。

一句话总结：**MMAUD 正常收尾即可；AIQII 先做第 1 步可视化，再查框尺寸和 anchor，基本就能找到根因。模型代码这次不用动。**
